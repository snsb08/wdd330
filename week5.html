Chapter 11 Testing and Debugging

Testing and Debugging is important for programmers because there is always a possible error or bug in a program. A programmer needs to constantly be looking for possible error and find ways to minimized them.

Possible causes of errors:

  • System error: the program is used in a system or device that is having problems. It is not the programmer's fault but by being aware of possible common system errors, problems can be reduced.
  • Programmer error: a typo can cause incorrect syntax or faulty logic.
  • User error: typing incorrect type of data can cause a problem. Programmers cannot control what the user will do but they can imagine ahead of time different possible user errors and make the program in a way that minimizes them.

It is better to know when an error occurs to be able to fix it, so programs must have alerts for error and warnings.

Feature Detection

It is impossible to know which browser will be used by each user, so the program has to find a way to be compatible and it can be done using feature detection. We can use an if statement that will prevent errors if certain methods are not supported by the browser being used. "Feature detection guarantees that the method is only called if it actually exists and fails gracefully, without any exceptions being thrown, if the method doesn't exist.

Debugging in the Browser

The Trusty alert() method can be used to debug a program by acting like breakpoint since the program stops until the ok is clicked.

Console.trace() method will log an interactive stack trace in the console. This will show functions that were called in the lead up to an exception occurring while the code is running.

Debugging tools

Most modern browsers have a debugging tool that sets breakpoints to check the code.

Error Objects

An error object can be created by the host environment when an exception occurs, or it can be created in the code using a constructor function.
const error = new Error('Oops, something went wrong');

There are seven more error objects used for specific errors:

  • EvalError is not used in the current ECMAScript specification and only retained for backwards compatibility. It was used to identify errors when using the global eval() function.
  • RangeError is thrown when a number is outside an allowable range of values.
  • ReferenceError is thrown when a reference is made to an item that doesn't exist. For example, calling a function that hasn't been defined.
  • SyntaxError is thrown when there's an error in the code’s syntax.
  • TypeError is thrown when there’s an error in the type of value used; for example, a string is used when a number is expected.
  • URIError is thrown when there’s a problem encoding or decoding the URI.
  • InternalError is a non-standard error that is thrown when an error occurs in the JavaScript engine. A common cause of this too much recursion.

All error objects have a number of properties, but they’re often used inconsistently across browsers. The only properties that are generally safe to use are:

  • The name property returns the name of the error constructor function used as a string, such as 'Error' or 'ReferenceError'.
  • The message property returns a description of the error and should be provided as an argument to the Error constructor function.
  • The stack property will return a stack trace for that error. This is a non-standard property and it’s recommended that it is not safe to use in production sites.

Throwing Exceptions

The throw statement can be applied to any JavaScript expression, causing the execution of the program to stop.

Exception Handling

Try: code in a try block will run as normal but if an exception occurs it will pass the error object that is thrown onto a catch block. Catch: code in a catch block will only run if an exception I thrown inside the try block. Finally: the finally block always executed after the try or catch block. It is used if you want some code to run in both cases.

Tests

Testing helps to write better code where errors can be identified early on.

Test-driven development (TDD) is considered to be best practice, but many do not use it.

  1. Write tests (that initially fail)
  2. Write code to pass the tests
  3. Refactor the code
  4. Test refactored code
  5. Write more tests for new features
